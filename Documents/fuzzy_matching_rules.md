# aiCompare 智慧對齊與模糊匹配規則說明文件 (Fuzzy Matching & Alignment Rules)

本文件詳細記載 `aiCompare` 中內建的智慧對齊、模糊檔名匹配、以及同目錄備份自我交叉對齊的核心演算法與業務規則。

---

## 1. Sorensen-Dice 檔名相似度演算法 (Sorensen-Dice Coefficient)

針對跨目錄比對時的「孤兒檔案（Orphans）」，系統採用經典的 **Sorensen-Dice 相似度係數** 來判定是否為「改名檔案」並進行自動對齊。

### 1.1 演算法原理
Sorensen-Dice 係數衡量兩個字串的字元雙元組（Bigrams）集合的交集比例：

$$\text{Similarity}(A, B) = \frac{2 \times |Bigrams(A) \cap Bigrams(B)|}{|Bigrams(A)| + |Bigrams(B)|}$$

- **Bigram 拆解範例**：
  - 字串 `"hello"` 拆解為 `{"he", "el", "ll", "lo"}`。
  - 字串 `"helo"` 拆解為 `{"he", "el", "lo"}`。
  - 交集為 `{"he", "el", "lo"}` (共 3 個)，相似度為 $\frac{2 \times 3}{4 + 3} \approx 0.857$。
- **特殊邊界處理**：
  - 若兩字串均為空字串，相似度為 `1.0`。
  - 若其中一個為空，相似度為 `0.0`。
  - 若字串長度過短無法生成 Bigrams（例如單個字元），降階為單字元匹配比例。

### 1.2 自動改名對齊規則 (Fuzzy Rename Match)
- **類別限制**：僅限相同類型的節點比對（資料夾對資料夾、檔案對檔案）。
- **閥值門檻**：相似度得分必須 **$\ge 0.6$ (60%)** 才會判定為改名匹配。
- **最佳配對優先**：當左側孤兒有多個潛在右側對齊目標時，選擇得分最高的配對。
- **狀態標記**：
  - 對齊後的行狀態標記為 `MODIFIED`。
  - `isRenamedMatch` 設為 `true`。
  - `matchConfidence` 設為 Sorensen-Dice 計算得分。
  - UI 雙欄將呈現 `left_name ➔ right_name` 的形式。

## 2. 同目錄備份自我交叉對齊 (Self-Cross Backup Alignment)

當使用者在左右雙欄選擇「同一個資料夾路徑」進行比對時，傳統比對會使每個檔案與自己對齊（狀態為 UNCHANGED），導致備份檔與原檔分散在不同行，無法進行內容對比。本機制旨在解決此痛點。

### 2.1 觸發條件
- 檢測到 `leftFolderPath == rightFolderPath` 時自動啟動。

### 2.2 備份後綴正則表達式 (Backup Suffix Regex Rules)
系統會自動掃描目錄下的檔案，利用正則表達式判定是否為備份檔，並解析出其對應的「原版主檔案」名稱：

1. **結尾型備份後綴** (如 `.old`, `.bak`, `.copy`)：
   - 匹配樣式：`^(.+)\.(old|bak|copy)$` (不區分大小寫)
   - 範例：`hello.c.old` ➔ 尋找原檔 `hello.c`
2. **版本號型後綴 (帶副檔名)** (如 `_v1`, `-v2` 置於副檔名前)：
   - 匹配樣式：`^(.+)[_-]v\d+\.([^.]+)$` (不區分大小寫)
   - 範例：`hello_v1.c` ➔ 尋找原檔 `hello.c`
3. **版本號型後綴 (無副檔名)** (如 `_v1` 置於檔尾)：
   - 匹配樣式：`^(.+)[_-]v\d+$` (不區分大小寫)
   - 範例：`hello_v1` ➔ 尋找原檔 `hello`
4. **複本標記型後綴 (帶副檔名)** (如 `-copy`, `_copy` 置於副檔名前)：
   - 匹配樣式：`^(.+)[_-]copy\.([^.]+)$` (不區分大小寫)
   - 範例：`hello-copy.c` ➔ 尋找原檔 `hello.c`
5. **複本標記型後綴 (無副檔名)** (如 `-copy` 置於檔尾)：
   - 匹配樣式：`^(.+)[_-]copy$` (不區分大小寫)
   - 範例：`hello-copy` ➔ 尋找原檔 `hello`
6. **系統預設複本樣式** (Windows 複製檔案時產生的 ` - Copy`)：
   - 匹配樣式：`^(.+)\s-\sCopy\.([^.]+)$` (不區分大小寫)
   - 範例：`hello - Copy.c` ➔ 尋找原檔 `hello.c`

### 2.3 重組與對齊行為 (Virtual Reconstitution)
- 當發現一對 `(原版主檔案, 備份檔案)` 且兩者均存在於同目錄中時，將其合併為單一 `FolderDiffRow`。
- **屬性整合**：
  - `leftFile` 填入原檔的 VFS 節點。
  - `rightFile` 填入備份檔的 VFS 節點。
  - `status` 設為 `FolderDiffStatus.MODIFIED`。
  - `isBackupMatch` 設為 `true`。
- 原本單獨對齊自己的兩個獨立行會被移除，改由此虛擬重組行取代，以確保左邊顯示原檔、右邊顯示備份檔。

### 2.4 UI 專屬呈現
- **文字顏色**：雙欄文字統一採用低調且具科技感的青藍色 (`0xFF569CD6`)。
- **中央控制軌道**：不顯示普通的 `≠`，而是顯示代表歷史版本回溯的**黃色「↩」圖示**（顏色值 `0xFFFFC107`）。
- **動作關聯**：雙擊該行（包含雙擊中央 Box），會直接開啟兩檔案的雙欄文字 Diff 頁籤，進行實際內容比對。

## 3. 手動拖曳對齊邏輯 (Manual Drag-and-Drop)

提供給使用者極大自由度，手動糾正任何演算法無法涵蓋的特殊對齊場景。

### 3.1 操作規則
- **拖曳源（Drag Source）**：必須是狀態為 `FolderDiffStatus.LEFT_ORPHAN` 的行。
- **置放目標（Drop Target）**：必須是狀態為 `FolderDiffStatus.RIGHT_ORPHAN`，且類型與拖曳源一致的行（資料夾對資料夾、檔案對檔案）。

### 3.2 互動回饋
- 拖曳開始時，該行會套用 `0.5f` 的不透明度（Alpha），呈現虛擬懸浮感。
- 拖曳中的源側（Left Column）顯示綠色邊框 (`0xFF4CAF50`)。
- 當滑鼠移經合法的右側目標時，目標行會立即亮起代表 drop 準備就緒的紫色邊框 (`0xFF9C27B0`) 與背景，提供出色的視覺引導。

## 4. 樹狀目錄排序原則 (Tree Sorting Order)

為了保持目錄樹視覺的精緻感與直覺性，系統在每一層目錄比對完畢後，強制執行以下排序：
1. **資料夾優先**：所有匹配的資料夾對、模糊資料夾對、以及孤兒資料夾排在最前面。
2. **檔案置後**：所有檔案排在資料夾的後方。
3. **組內字母排序**：同類型組別內部（如資料夾與資料夾之間、檔案與檔案之間）一律依檔名進行 `A-Z` 字母排序。
